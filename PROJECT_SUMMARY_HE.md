# סיכום פרויקט Manifesto

## מהו הפרויקט?

**Manifesto** היא מערכת Web פנימית לניהול לוגיסטיקה, מלאי וספקים, המשולבת עם יכולות חיפוש ושיחה מבוססות בינה מלאכותית.

שם המערכת נלקח מהמונח *Cargo Manifest* — מסמך המתאר את תכולת המשלוח, מקורו והספק ששלח אותו. מטרת המוצר היא ליצור מקור מידע מרכזי, מסודר וניתן לחיפוש עבור כל המוצרים, הספקים, המשלוחים ומסמכי המדיניות של הארגון.

המערכת מיועדת לשלושה סוגי משתמשים:

| תפקיד | יכולות מרכזיות |
|---|---|
| עובד | שאילת שאלות על נהלים ומדיניות ארגונית באמצעות צ'אט RAG, כולל הפניות למסמכי המקור |
| מנהל | צפייה וניהול של מלאי, מוצרים, ספקים ומשלוחים, לצד שאילת שאלות בשפה טבעית על הנתונים |
| מנהל מערכת | כל יכולות המנהל, וכן יצירה וניהול של חשבונות משתמשים |

המערכת מחולקת לשני אזורים עיקריים:

1. **ניהול מלאי ולוגיסטיקה** — חיפוש מוצרים, מעקב אחר ספקים, צפייה בהיסטוריית משלוחים והוספת פריטים חדשים.
2. **שיחה מבוססת AI** — מענה על שאלות בנוגע למדיניות הארגון ולנתוני הלוגיסטיקה באמצעות שפה טבעית.

## יכולות AI ו־RAG

הפרויקט מתוכנן לתמוך בשני תהליכי RAG נפרדים:

- **Policy RAG לעובדים** — קליטת מסמכי PDF ו־DOCX, חלוקתם למקטעים, יצירת embeddings, חיפוש וקטורי והחזרת תשובה עם ציטוט מקורות.
- **Logistics RAG למנהלים** — מענה על שאלות בשפה טבעית בהתבסס על נתוני המלאי והמשלוחים הנמצאים ב־PostgreSQL.

שכבת `LLMService` מספקת ממשק אחיד למודלי שפה, כדי לאפשר שימוש ב־Ollama בהרצה מקומית וב־OpenAI בסביבת ענן, ללא שינוי הלוגיקה העסקית.

## ה־Stack הטכנולוגי

### Backend

| טכנולוגיה | שימוש |
|---|---|
| Python 3.12 | שפת הפיתוח של השרת |
| FastAPI | API אסינכרוני, ולידציה ותיעוד OpenAPI |
| SQLAlchemy Async | שכבת ORM וגישה אסינכרונית למסד הנתונים |
| asyncpg | דרייבר PostgreSQL אסינכרוני |
| Alembic | ניהול migrations |
| Pydantic Settings | ניהול הגדרות ומשתני סביבה |
| PyJWT ו־Passlib/Bcrypt | אימות משתמשים, JWT והצפנת סיסמאות |
| PyMuPDF ו־python-docx | קריאת מסמכי PDF ו־DOCX |
| Structlog | לוגים מובנים |

### Frontend

| טכנולוגיה | שימוש |
|---|---|
| React 18 | בניית ממשק המשתמש |
| TypeScript | טיפוסים ובטיחות בזמן פיתוח |
| Vite | סביבת פיתוח ותהליך build |
| Tailwind CSS | עיצוב הממשק |
| React Router | ניווט ונתיבים מוגנים לפי תפקיד |
| Zustand | ניהול state בצד הלקוח |
| Axios | תקשורת מול ה־API |
| TanStack React Query | ניהול מידע המגיע מהשרת, cache ומצבי טעינה |

### תשתיות ונתונים

| טכנולוגיה | שימוש |
|---|---|
| PostgreSQL 16 | מסד הנתונים המרכזי |
| pgvector | אחסון embeddings וחיפוש וקטורי |
| Ollama | הרצת מודלי שפה מקומיים |
| OpenAI API | ספק מודלי שפה בענן |
| Docker ו־Docker Compose | הרצת מסד הנתונים, השרת ו־Ollama בסביבה אחידה |

בפיתוח מקומי, ה־Backend זמין בפורט `8000`, PostgreSQL בפורט `5432`, ו־Ollama בפורט `11434`. ה־Frontend רץ באמצעות Vite ומעביר בקשות API ל־FastAPI.

## אבטחה והרשאות

- ההזדהות מבוססת JWT.
- תפקיד המשתמש נשמר בתוך ה־token ונבדק בכל נתיב API מוגן.
- נתיבי ה־Frontend מוגנים ומוצגים בהתאם לתפקיד המשתמש.
- צ'אט הלוגיסטיקה מתוכנן להשתמש בחיבור PostgreSQL לקריאה בלבד.
- שאילתות SQL שייווצרו באמצעות מודל שפה יאושרו רק אם הן שאילתות `SELECT`.
- סודות והגדרות סביבתיות מנוהלים באמצעות קובצי `.env`.

### עקרונות העבודה

1. **בעלות תחומית** — כל סוכן משנה רק קבצים בתחום האחריות שלו.
2. **שינוי ממוקד בכל commit** — כל commit מטפל בנושא אחד ונמצא בבעלות סוכן אחד.
3. **תזמור מרכזי** — כל התקשורת והעברת המידע בין הסוכנים עוברת דרך Claude.
4. **אישור אנושי** — מוביל הצוות מאשר את תחילת העבודה ואת ה־commit לפני שהוא נכנס להיסטוריה.
5. **הקשר מינימלי ומדויק** — כל סוכן מקבל רק את המפרט, ההחלטות והקבצים הנדרשים למשימה שלו.
6. **בדיקות לפני commit** — שינוי אינו מתקדם עד שמעבר הבדיקות הרלוונטיות הושלם בהצלחה.
7. **תיעוד מתמשך** — החלטות, ארכיטקטורה, מצב הפרויקט, צריכת tokens והעברות אחריות מתועדים בקבצים ייעודיים.
